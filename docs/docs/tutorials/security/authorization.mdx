Authorization in the Model Context Protocol (MCP) secures access to sensitive resources and
operations exposed by MCP servers. If your MCP server handles user data or administrative
actions, authorization ensures only permitted users can access its endpoints.

MCP doesn't require a specific identity provider or auth system. Instead, it builds on
[OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1-13) conventions to let clients
and servers establish trust in a consistent way. For the full details, see the
[Authorization specification](/specification/latest/basic/authorization).

## When Should You Use Authorization?

While authorization for MCP servers is **optional**, it is strongly recommended when:

- Your server works with user-specific data like emails, documents, or databases
- You need to track which user performed a given action
- Your server exposes APIs that require user consent before access
- You're targeting enterprise environments with strict access controls
- You want to rate-limit or track usage on a per-user basis

<Tip>
**Authorization for Local MCP Servers**

For MCP servers using the [STDIO transport](/specification/latest/basic/transports#stdio), you
can use environment-based credentials or credentials provided by third-party libraries embedded
directly in the MCP server instead. Because a STDIO-built MCP server runs locally, it has
access to a range of flexible options when it comes to acquiring user credentials that may or
may not rely on in-browser authentication and authorization flows.

OAuth flows, in turn, are designed for HTTP-based transports where the MCP server is
remotely-hosted and the client uses OAuth to establish that a user is authorized to access
said remote server.

</Tip>

## The Authorization Flow: Step by Step

Let's walk through what happens when a client wants to connect to your protected MCP server:

<Steps>
<Step title="Initial Handshake">
When your MCP client first tries to connect, your server responds with a `401 Unauthorized`
and tells the client where to find authorization information, captured in a
[Protected Resource Metadata (PRM) document](https://datatracker.ietf.org/doc/html/rfc9728).
The document is hosted by the MCP server, follows a predictable path pattern, and is provided
to the client in the `resource_metadata` parameter within the `WWW-Authenticate` header.

```http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="mcp",
  resource_metadata="https://your-server.com/.well-known/oauth-protected-resource",
  scope="mcp:tools mcp:resources"
```

This tells the client that authorization is required for the MCP server, where to get the
necessary information to kickstart the authorization flow, and which scopes are needed to
access the resource. For more on how clients should select scopes, see the
[Scope Selection Strategy](/specification/latest/basic/authorization#scope-selection-strategy)
section in the authorization spec.

</Step>

<Step title="Protected Resource Metadata Discovery">
With the URI pointer to the PRM document, the client will fetch the metadata to learn about
the authorization server, supported scopes, and other resource information. The data is
typically encapsulated in a JSON blob, similar to the one below.

```json
{
  "resource": "https://your-server.com/mcp",
  "authorization_servers": ["https://auth.your-server.com"],
  "scopes_supported": ["mcp:tools", "mcp:resources"]
}
```

You can see a more comprehensive example in
[RFC 9728 Section 3.2](https://datatracker.ietf.org/doc/html/rfc9728#name-protected-resource-metadata-r).

</Step>

<Step title="Authorization Server Discovery">
Next, the client discovers what the authorization server can do by fetching its metadata.
If the PRM document lists more than one authorization server, the client can decide which
one to use.

With an authorization server selected, the client will then construct a standard metadata URI
and issue a request to the
[OpenID Connect (OIDC) Discovery](https://openid.net/specs/openid-connect-discovery-1_0.html)
or [OAuth 2.0 Auth Server Metadata](https://datatracker.ietf.org/doc/html/rfc8414) endpoints
(depending on authorization server support) and retrieve another set of metadata properties
that will allow it to know the endpoints it needs to complete the authorization flow.

```json
{
  "issuer": "https://auth.your-server.com",
  "authorization_endpoint": "https://auth.your-server.com/authorize",
  "token_endpoint": "https://auth.your-server.com/token",
  "registration_endpoint": "https://auth.your-server.com/register",
  "client_id_metadata_document_supported": true
}
```

</Step>

<Step title="Client Registration">
With all the metadata out of the way, the client now needs to make sure that it's registered
with the authorization server. MCP supports three client registration mechanisms:

| Approach                         | When to Use                                                  |
| -------------------------------- | ------------------------------------------------------------ |
| **Client ID Metadata Documents** | When client and server have no prior relationship            |
| **Pre-registration**             | When client and server have an existing relationship         |
| **Dynamic Client Registration**  | For backwards compatibility with older authorization servers |

See [Client Registration Approaches](#client-registration-approaches) for detailed
information on each mechanism.

</Step>

<Step title="User Authorization">
The client will now need to open a browser to the `/authorize` endpoint, where the user can
log in and grant the required permissions. The authorization server will then redirect back
to the client with an authorization code that the client exchanges for tokens:

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIs...",
  "refresh_token": "def502...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

The access token is what the client will use to authenticate requests to the MCP server.
This step follows standard
[OAuth 2.1 authorization code with PKCE](https://oauth.net/2/grant-types/authorization-code/)
conventions.

</Step>

<Step title="Making Authenticated Requests">
Finally, the client can make requests to your MCP server using the access token embedded in
the `Authorization` header:

```http
GET /mcp HTTP/1.1
Host: your-server.com
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
```

The MCP server will need to validate the token and process the request if the token is valid
and has the required permissions.

</Step>
</Steps>

## Client Registration Approaches

MCP supports three client registration mechanisms to accommodate different deployment scenarios:

### Client ID Metadata Documents

<Note>This is the recommended approach for client registration in MCP.</Note>

Client ID Metadata Documents
([draft-ietf-oauth-client-id-metadata-document-00](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-client-id-metadata-document-00))
allow MCP clients to use an HTTPS URL as their `client_id`. The authorization server fetches
client metadata from this URL, eliminating the need for pre-registration or dynamic registration.

**How it works:**

1. The MCP client hosts a JSON metadata document at a publicly accessible HTTPS URL
2. When starting authorization, the client uses this URL as its `client_id`
3. The authorization server detects the URL format and fetches the metadata document
4. The server validates the document and uses the metadata (like `redirect_uris`) to process
   the authorization request

**Advantages:**

- No pre-existing relationship required between client and server
- Client maintains full control over its metadata
- No server-side storage needed for client registration
- Metadata can be updated by the client at any time

**Requirements for clients:**

- Host metadata at an HTTPS URL with a path component
  (e.g., `https://example.com/oauth/metadata.json`)
- Include required fields: `client_id` (must match the URL), `client_name`, `redirect_uris`
- Ensure the document is publicly accessible

### Pre-Registration

Pre-registration is appropriate when the client and authorization server have an existing
relationship. The client can have embedded client registration information (client ID and
optionally client credentials) that it uses to complete the authorization flow.

### Dynamic Client Registration

<Warning>
  This is a legacy approach maintained for backwards compatibility with earlier
  versions of the MCP authorization specification. New implementations should
  prefer Client ID Metadata Documents when the authorization server supports
  them.
</Warning>

Dynamic Client Registration
([RFC7591](https://datatracker.ietf.org/doc/html/rfc7591)) allows clients to programmatically
register with an authorization server by sending a registration request containing client
metadata.

**When to use DCR:**

- The authorization server doesn't support Client ID Metadata Documents
- You're integrating with legacy systems
- The authorization server requires server-side client registration for policy reasons

**Security considerations:**

- Unauthenticated DCR means anyone can register any client with the authorization server
- Authorization servers should implement appropriate controls (trusted hosts, rate limiting,
  etc.)

### Fallback Priority

For the recommended priority order when clients support multiple registration options, see the
[Client Registration Approaches](/specification/latest/basic/authorization#client-registration-approaches)
section in the authorization spec.

## Implementing Client Registration

The sections below walk through setting up end-to-end authorization — including the MCP
server, identity provider, and client registration — using the two most common approaches:
Client ID Metadata Documents (CIMD) and Dynamic Client Registration (DCR).

### Client ID Metadata Documents with Stytch

<Note>
  This example uses [Stytch](https://stytch.com/) as the identity provider
  because it supports CIMD natively through its [Connected
  Apps](https://stytch.com/docs/connected-apps) feature. The same approach works
  with any CIMD-capable identity provider — the key requirement is that your
  authorization server advertises `"client_id_metadata_document_supported":
  true` in its metadata.
</Note>

CIMD lets MCP clients use an HTTPS URL as their `client_id`. The authorization server fetches
client metadata from that URL at authorization time, so there's no need for pre-registration
or dynamic registration. Here's what it takes to get this working end-to-end.

#### Stytch Setup

Create a [Stytch](https://stytch.com/) account and set up a new project. In the Stytch
Dashboard, navigate to **Connected Apps → Settings** and
[enable CIMD support](https://stytch.com/docs/connected-apps/oauth-learn-more/client-types#client-id-metadata-documents-beta).
This tells Stytch to accept URL-based `client_id` values and fetch client metadata from those
URLs during authorization. Without this setting enabled, Stytch will reject URL-based
`client_id` values with "The Connected App requested could not be found."

While in **Connected Apps → Settings**, also set the **Authorization URL** to the location
in your web app where the Stytch `<IdentityProvider>` component is hosted (e.g.,
`http://localhost:3002/authorize`). This URL is served in your project's OpenID Configuration
endpoint and is where users will be redirected to authenticate.

  />
</Frame>

Next, [register any custom scopes](https://stytch.com/docs/connected-apps/oauth-learn-more/scopes)
your MCP server needs. Navigate to **Connected Apps → Scopes** and add a `mcp:tools` scope.
This scope will be required for clients to access tools on your MCP server.

Finally, go to **Frontend SDKs → Authorized applications** and add the domain where your

#### Write the MCP Server

Your MCP server acts as an OAuth 2.1 resource server — it validates access tokens and serves
Protected Resource Metadata so clients know where to authenticate. The SDK handles metadata
endpoints and `WWW-Authenticate` challenges for you.

For CIMD, MCP clients check the Authorization Server metadata for
`client_id_metadata_document_supported: true` to decide whether to use CIMD or fall back to
DCR. How this flag gets served depends on the SDK.

<Tabs>
<Tab title="TypeScript">

In TypeScript, the MCP server proxies OAuth endpoints to Stytch using
`ProxyOAuthServerProvider`. The SDK's default metadata helper doesn't include the CIMD flag,
so you serve the AS metadata yourself before `mcpAuthRouter`:

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StreamableHTTPServerTransport } from "@modelcontextprotocol/sdk/server/streamableHttp.js";
import { createMcpExpressApp } from "@modelcontextprotocol/sdk/server/express.js";
import {
  mcpAuthRouter,
  createOAuthMetadata,
} from "@modelcontextprotocol/sdk/server/auth/router.js";
import { ProxyOAuthServerProvider } from "@modelcontextprotocol/sdk/server/auth/providers/proxyProvider.js";
import { createRemoteJWKSet, jwtVerify } from "jose";
import { z } from "zod";

const STYTCH_DOMAIN = "https://your-project.stytch.dev";
const SERVER_URL = new URL("http://localhost:3001");
const AUTH_PUBLIC_URL = "http://localhost:3002"; // Your authorize UI

const JWKS = createRemoteJWKSet(
  new URL(`${STYTCH_DOMAIN}/.well-known/jwks.json`),
);

const provider = new ProxyOAuthServerProvider({
  endpoints: {
    authorizationUrl: `${AUTH_PUBLIC_URL}/authorize`,
    tokenUrl: `${STYTCH_DOMAIN}/v1/oauth2/token`,
  },
  verifyAccessToken: async (token: string) => {
    const { payload } = await jwtVerify(token, JWKS, {
      issuer: STYTCH_DOMAIN,
    });
    return {
      token,
      clientId: (payload.sub as string) ?? "",
      scopes: typeof payload.scope === "string" ? payload.scope.split(" ") : [],
      expiresAt: payload.exp,
    };
  },
  getClient: async (clientId: string) => {
    // CIMD: fetch client metadata from the client_id URL
    try {
      const resp = await fetch(clientId, {
        headers: { Accept: "application/json" },
      });
      if (!resp.ok) return undefined;
      return await resp.json();
    } catch {
      return undefined;
    }
  },
});

const app = createMcpExpressApp();

// Serve AS metadata with CIMD support flag
const oauthMetadata = createOAuthMetadata({
  provider,
  issuerUrl: SERVER_URL,
  baseUrl: SERVER_URL,
  scopesSupported: ["mcp:tools"],
});
app.get("/.well-known/oauth-authorization-server", (_req, res) => {
  res.json({
    ...oauthMetadata,
    client_id_metadata_document_supported: true,
  });
});

// Serve protected resource metadata and proxy auth endpoints
app.use(
  mcpAuthRouter({
    provider,
    issuerUrl: SERVER_URL,
    baseUrl: SERVER_URL,
    resourceServerUrl: SERVER_URL,
    scopesSupported: ["mcp:tools"],
  }),
);

app.all("/mcp", async (req, res) => {
  const server = new McpServer({ name: "example-server", version: "1.0.0" });

  server.registerTool(
    "add",
    {
      title: "Addition Tool",
      description: "Add two numbers together",
      inputSchema: {
        a: z.number().describe("First number"),
        b: z.number().describe("Second number"),
      },
    },
    async ({ a, b }) => ({
      content: [{ type: "text", text: `${a} + ${b} = ${a + b}` }],
    }),
  );

  const transport = new StreamableHTTPServerTransport({
    sessionIdGenerator: undefined,
  });

  res.on("close", () => {
    transport.close().catch(() => {});
    server.close().catch(() => {});
  });

  await server.connect(transport);
  await transport.handleRequest(req, res, req.body);
});

app.listen(3001, () => {
  console.log("MCP server listening on http://localhost:3001/mcp");
});
```

The `getClient` callback is where CIMD happens — when a URL-formatted `client_id` is
received, it fetches the metadata document from that URL and validates it.

For more details, see the
[TypeScript SDK documentation](https://github.com/modelcontextprotocol/typescript-sdk).

</Tab>
<Tab title="C#">

In C#, the MCP server points clients directly to Stytch as the external authorization
server. The `AddMcp()` extension configures Protected Resource Metadata, and the
`McpAuthenticationHandler` automatically serves it and adds `WWW-Authenticate` headers on
`401` responses. The CIMD flag comes from Stytch's own AS metadata, not the MCP server.

```csharp
using Microsoft.AspNetCore.Authentication.JwtBearer;
using Microsoft.IdentityModel.Tokens;
using ModelContextProtocol.AspNetCore.Authentication;
using ModelContextProtocol.Server;
using System.ComponentModel;

var builder = WebApplication.CreateBuilder(args);

var serverUrl = "http://localhost:3001";
var stytchDomain = "https://your-project.stytch.dev";

builder.Services.AddAuthentication(options =>
{
    options.DefaultChallengeScheme = McpAuthenticationDefaults.AuthenticationScheme;
    options.DefaultAuthenticateScheme = JwtBearerDefaults.AuthenticationScheme;
})
.AddJwtBearer(options =>
{
    options.Authority = stytchDomain;
    options.TokenValidationParameters = new TokenValidationParameters
    {
        ValidIssuer = stytchDomain,
        ValidAudience = serverUrl,
    };
})
.AddMcp(options =>
{
    options.ResourceMetadata = new()
    {
        AuthorizationServers = { stytchDomain },
        ScopesSupported = ["mcp:tools"],
    };
});

builder.Services.AddAuthorization();
builder.Services.AddMcpServer()
    .WithTools<MathTools>()
    .WithHttpTransport();

var app = builder.Build();

app.UseAuthentication();
app.UseAuthorization();

app.MapMcp().RequireAuthorization();

app.Run(serverUrl);

[McpServerToolType]
public static class MathTools
{
    [McpServerTool]
    [Description("Add two numbers together")]
    public static string Add(double a, double b)
        => $"{a} + {b} = {a + b}";
}
```

For more details, see the
[C# SDK documentation](https://github.com/modelcontextprotocol/csharp-sdk).

</Tab>
</Tabs>

#### Testing with CIMD

With the server running, connect an MCP client (like
[Visual Studio Code](https://code.visualstudio.com)) to your MCP server endpoint at
`http://localhost:3001`. On connection, the authorization flow will proceed as follows:

1. The server returns a `401` with `WWW-Authenticate` pointing to the Protected Resource Metadata
2. The client fetches the PRM, discovers the AS metadata URL
3. The AS metadata advertises `client_id_metadata_document_supported: true`
4. The client uses CIMD and initiates the OAuth flow
5. A browser window opens to your authorization UI with the Stytch login
6. After authentication, the client receives a token and retries the tool call

  />
</Frame>

You can also use [client.dev](https://client.dev/) as a reference authorization server that
supports CIMD out of the box for quick testing and experimentation.

<Note>
**Common Issues**

- **Metadata document not accessible**: The authorization server must be able to fetch your
  `client_id` URL. Verify the URL is reachable from the AS by testing with `curl` or a
  browser from the server's network.
- **`client_id` mismatch**: The `client_id` field in the metadata document must exactly match
  the URL where the document is hosted. Even trailing slash differences will cause validation
  failures.
- **Missing required fields**: Ensure `client_name`, `redirect_uris`, and
  `token_endpoint_auth_method` are present in your metadata document.
- **HTTPS requirement**: The metadata document URL must use HTTPS (not HTTP) with a path
  component.
- **Scopes must be registered**: Custom scopes must be registered in your identity provider's
  dashboard. Without this, the IdP will reject the authorization request.

</Note>

#### Write the MCP Server

With Keycloak configured, you can now write the MCP server. The server needs the Keycloak
realm URL as its authorization server and must validate tokens against it. The `mcp:tools`
scope we created earlier will be required for access.

<Tabs>
<Tab title="TypeScript">

```typescript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StreamableHTTPServerTransport } from "@modelcontextprotocol/sdk/server/streamableHttp.js";
import { createMcpExpressApp } from "@modelcontextprotocol/sdk/server/express.js";
import {
  mcpAuthRouter,
  createOAuthMetadata,
} from "@modelcontextprotocol/sdk/server/auth/router.js";
import { ProxyOAuthServerProvider } from "@modelcontextprotocol/sdk/server/auth/providers/proxyProvider.js";
import { z } from "zod";

const KEYCLOAK_URL = "http://localhost:8080/realms/master";
const SERVER_URL = new URL("http://localhost:3000");

// Server client credentials (from Keycloak client registration)
const CLIENT_ID = process.env.OAUTH_CLIENT_ID || "mcp-server";
const CLIENT_SECRET = process.env.OAUTH_CLIENT_SECRET || "";

const provider = new ProxyOAuthServerProvider({
  endpoints: {
    authorizationUrl: `${KEYCLOAK_URL}/protocol/openid-connect/auth`,
    tokenUrl: `${KEYCLOAK_URL}/protocol/openid-connect/token`,
  },
  verifyAccessToken: async (token: string) => {
    // Validate token via Keycloak introspection endpoint
    const response = await fetch(
      `${KEYCLOAK_URL}/protocol/openid-connect/token/introspect`,
      {
        method: "POST",
        headers: { "Content-Type": "application/x-www-form-urlencoded" },
        body: new URLSearchParams({
          token,
          client_id: CLIENT_ID,
          client_secret: CLIENT_SECRET,
        }),
      },
    );

    const data = await response.json();
    if (!data.active) throw new Error("Token is not active");

    return {
      token,
      clientId: data.client_id,
      scopes: data.scope ? data.scope.split(" ") : [],
      expiresAt: data.exp,
    };
  },
});

const app = createMcpExpressApp();

app.use(
  mcpAuthRouter({
    provider,
    issuerUrl: SERVER_URL,
```

## Common Pitfalls and How to Avoid Them

For comprehensive security guidance, including attack vectors, mitigation strategies, and
implementation best practices, make sure to read through
[Security Best Practices](/specification/draft/basic/security_best_practices). A few key
issues are called out below.

- **Do not implement token validation or authorization logic by yourself**. Use off-the-shelf,
  well-tested, and secure libraries for things like token validation or authorization
  decisions. Doing everything from scratch means that you're more likely to implement things
  incorrectly unless you are a security expert.
- **Use short-lived access tokens**. Depending on the authorization server used, this setting
  might be customizable. We recommend to not use long-lived tokens - if a malicious actor
  steals them, they will be able to maintain their access for longer periods.
- **Always validate tokens**. Just because your server received a token does not mean that the
  token is valid or that it's meant for your server. Always verify that what your MCP server
  is getting from the client matches the required constraints.
- **Store tokens in secure, encrypted storage**. In certain scenarios, you might need to cache
  tokens server-side. If that is the case, ensure that the storage has the right access
  controls and cannot be easily exfiltrated by malicious parties with access to your server.
  You should also implement robust cache eviction policies to ensure that your MCP server is
  not re-using expired or otherwise invalid tokens.
- **Enforce HTTPS in production**. Do not accept tokens or redirect callbacks over plain HTTP
  except for `localhost` during development.
- **Least-privilege scopes**. Don't use catch‑all scopes. Split access per tool or capability
  where possible and verify required scopes per route/tool on the resource server.
- **Don't log credentials**. Never log `Authorization` headers, tokens, codes, or secrets.
  Scrub query strings and headers. Redact sensitive fields in structured logs.
- **Separate app vs. resource server credentials**. Don't reuse your MCP server's client
  secret for end‑user flows. Store all secrets in a proper secret manager, not in source
  control.
- **Return proper challenges**. On 401, include `WWW-Authenticate` with `Bearer`, `realm`,
  and `resource_metadata` so clients can discover how to authenticate.
- **DCR (Dynamic Client Registration) controls**. If using the legacy DCR approach, be aware
  of constraints specific to your organization, such as trusted hosts, required vetting, and
  audited registrations. Unauthenticated DCR means that anyone can register any client with
  your authorization server. Consider migrating to Client ID Metadata Documents for improved
  security and flexibility.
- **Multi‑tenant/realm mix-ups**. Pin to a single issuer/tenant unless explicitly
  multi‑tenant. Reject tokens from other realms even if signed by the same authorization
  server.
- **Audience/resource indicator misuse**. Don't configure or accept generic audiences (like
  `api`) or unrelated resources. Require the audience/resource to match your configured
  server.
- **Error detail leakage**. Return generic messages to clients, but log detailed reasons with
  correlation IDs internally to aid troubleshooting without exposing internals.
- **Session identifier hardening**. Treat `Mcp-Session-Id` as untrusted input; never tie
  authorization to it. Regenerate on auth changes and validate lifecycle server‑side.

## Related Standards and Documentation

MCP authorization builds on these well-established standards:

- **[OAuth 2.1](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1-13)**: The core
  authorization framework
- **[RFC 8414](https://datatracker.ietf.org/doc/html/rfc8414)**: Authorization Server Metadata
  discovery
- **[OAuth Client ID Metadata Document](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-client-id-metadata-document-00)**:
  Client ID Metadata Documents
- **[RFC 7591](https://datatracker.ietf.org/doc/html/rfc7591)**: Dynamic Client Registration
- **[RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728)**: Protected Resource Metadata
- **[RFC 8707](https://datatracker.ietf.org/doc/html/rfc8707)**: Resource Indicators

For additional details, refer to:

- [Authorization Specification](/specification/latest/basic/authorization)
- [Security Best Practices](/specification/latest/basic/security_best_practices)
- [Available MCP SDKs](/docs/sdk)

Understanding these standards will help you implement authorization correctly and
troubleshoot issues when they arise.
