If you run an MCP server, the protocol underneath it is about to take its
biggest step since launch. The
[`2026-07-28` specification release candidate](/posts/2026-07-28-release-candidate/)
removes the `initialize` handshake and the protocol-level session, completing
the plan we laid out in
[The Future of MCP Transports](/posts/2025-12-19-mcp-transport-future/). Your
server becomes something a plain round-robin load balancer can scale, with no
sticky sessions and no shared session store.

Today you can try that for real: the official
[Python](https://github.com/modelcontextprotocol/python-sdk) and
[TypeScript](https://github.com/modelcontextprotocol/typescript-sdk) SDKs
now have betas that speak `2026-07-28`. The specification goes final on
July 28, 2026. That leaves four weeks to run the betas against your real
workloads and tell us what breaks while changes are still cheap.

The [Go](https://github.com/modelcontextprotocol/go-sdk) and
[C#](https://github.com/modelcontextprotocol/csharp-sdk) SDKs will also ship
beta releases with `2026-07-28` support. Watch their repositories for the
announcements.

## Your existing server keeps working

First, the reassurance. Nothing breaks today, and nothing breaks on July 28
either. That date is when the specification text is published, not a
switch-off for servers on the current protocol:

- The stable v1.x lines remain the recommended versions for production.
  Installing either SDK without an explicit version pin still resolves to
  v1.x. After v2 ships, the TypeScript SDK will keep shipping v1.x bug
  fixes and security updates for at least six months, and the Python v1.x
  line continues to receive critical bug fixes and security patches.
- Clients that speak `2026-07-28` fall back to the `initialize` handshake
  when they reach a server on `2025-11-25` or earlier, so old servers and new
  clients keep interoperating.
- If you publish a library that depends on the Python `mcp` package, add an
  upper bound now (for example `mcp>=1.27,<2`) so the stable v2 release does
  not surprise your users.

The betas are opt-in by design, and that extends to the wire: in the
TypeScript SDK, upgrading to v2 does not by itself change what your server
speaks. Serving `2026-07-28` is an explicit choice you make when you wire up
the transport.

## Python: `mcp` 2.0

The v2 line is a rework of the package you already know, and the API got
smaller in the process. A complete server:

```python
from mcp.server import MCPServer

mcp = MCPServer("Demo")


@mcp.tool()
def add(a: int, b: int) -> int:
    """Add two numbers."""
    return a + b


@mcp.resource("greeting://{name}")
def greeting(name: str) -> str:
    """Greet someone by name."""
    return f"Hello, {name}!"
```

No JSON Schema to write: `a: int, b: int` is the schema. Install the beta
with an exact pin, since unpinned installs stay on v1.x:

```bash
uv add "mcp[cli]==2.0.0b1"
# or
pip install "mcp[cli]==2.0.0b1"
```

If you have a v1 server today, the
[migration guide](https://py.sdk.modelcontextprotocol.io/v2/migration/)
walks through every breaking change. The ones you will hit first:

- `FastMCP` is renamed to `MCPServer`. Same class, new name.
- Transport settings like `host` and `port` move from the constructor to
  `run()` and the app factory methods.
- Model fields are snake_case in Python now (`result.is_error`, not
  `result.isError`). The JSON wire format is unchanged.
- The wire types live in a standalone
  [`mcp-types`](https://pypi.org/project/mcp-types/) package that depends
  only on `pydantic` and `typing-extensions`, so lightweight tooling no
  longer drags in the whole transport stack.

On the protocol side, the beta speaks `2026-07-28` end to end: servers
answer the new `server/discover` method, and the client's default mode
probes it and falls back to `initialize` on older servers. Upgrading your
server does not strand existing clients: a v2 server answers the legacy
`initialize` handshake alongside `server/discover`, so clients on
`2025-11-25` keep connecting. You can test all
of it in-process: pass your `MCPServer` instance directly to `Client`
(`async with Client(mcp) as client:`) and it connects in memory, the same
pattern as FastAPI's `TestClient`. No subprocess, no port. The [SDK documentation](https://py.sdk.modelcontextprotocol.io/v2/)
has the tutorial and the full API reference.

## TypeScript: split packages

TypeScript v2 retires the monolithic `@modelcontextprotocol/sdk` package in
favor of focused ones: `@modelcontextprotocol/server` for servers,
`@modelcontextprotocol/client` for clients, plus thin adapters for Node.js,
Express, Hono, and Fastify. It is ESM-only and runs on Node.js 20+, Bun, and
Deno. Tool schemas now use
[Standard Schema](https://standardschema.dev/), so you can bring Zod v4,
Valibot, ArkType, or any compatible library:

```typescript
import { McpServer } from "@modelcontextprotocol/server";
import { StdioServerTransport } from "@modelcontextprotocol/server/stdio";
import * as z from "zod/v4";

const server = new McpServer({ name: "greeting-server", version: "1.0.0" });

server.registerTool(
  "greet",
  {
    description: "Greet someone by name",
    inputSchema: z.object({ name: z.string() }),
  },
  async ({ name }) => ({
    content: [{ type: "text", text: `Hello, ${name}!` }],
  }),
);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
}

main();
```

```bash
npm install @modelcontextprotocol/server@beta
npm install @modelcontextprotocol/client@beta
```

The stdio example is the smallest possible server. For HTTP deployments,
which is where the stateless protocol pays off, `createMcpHandler` from
`@modelcontextprotocol/server` is the entry point: it serves `2026-07-28`
per request and handles `2025-11-25` traffic from the same endpoint, with
the adapter packages connecting it to Node.js, Express, Hono, and Fastify.

Migrating from v1 is mostly mechanical, and there is a codemod that does the
mechanical part for you, including the rename from `.tool()` to
`registerTool` and the error-type renames:

```bash
npx @modelcontextprotocol/codemod@beta v1-to-v2 .
```

The repository ships two guides:
[upgrading from v1 to v2](https://github.com/modelcontextprotocol/typescript-sdk/blob/main/docs/migration/upgrade-to-v2.md),
and
[adopting the 2026-07-28 revision](https://github.com/modelcontextprotocol/typescript-sdk/blob/main/docs/migration/support-2026-07-28.md)
once you are on v2. They are separate steps on purpose: you can move to v2
now and turn on the new protocol revision when you are ready. The
[v2 API documentation](https://ts.sdk.modelcontextprotocol.io/v2/) covers
the rest.

## What the betas implement

Both SDKs implement the protocol changes from the release candidate. The
[RC announcement](/posts/2026-07-28-release-candidate/) covers them in
depth; the short version:

- **The stateless core**
  ([SEP-2575](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2575),
  [SEP-2567](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2567)).
  Every request is self-describing, capabilities come from
  `server/discover`, and any server instance can handle any request.
- **Multi round-trip requests**
  ([SEP-2322](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2322)).
  Tools can return `InputRequiredResult` to ask the user something
  mid-call, and the client retries with the answers. No long-lived stream
  required.
- **Routable transport headers**
  ([SEP-2243](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2243)).
  `Mcp-Method` and `Mcp-Name` ride on every request so gateways and rate
  limiters can route without parsing bodies.
- **Authorization hardening**, including `iss` validation per
  [RFC 9207](https://www.rfc-editor.org/rfc/rfc9207)
  ([SEP-2468](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2468)),
  `application_type` in Dynamic Client Registration
  ([SEP-837](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/837)),
  scope accumulation on step-up
  ([SEP-2350](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2350)),
  and credential binding to the issuing authorization server
  ([SEP-2352](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2352)).
  The `application_type` change is the one desktop and CLI clients will
  feel: authorization servers stop defaulting them to `"web"` and
  rejecting their localhost redirects.
- **Standard error codes**
  ([SEP-2164](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2164)).
  A missing resource now returns JSON-RPC `-32602` instead of the
  MCP-custom `-32002`. If your client matches on the literal value, update
  it.
- **Deprecation annotations**
  ([SEP-2577](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2577))
  on roots, sampling, and logging. Advisory only; the deprecated methods,
  types, and capability flags keep working in this release and in every
  specification version published within a year of it.

The full protocol text is in the
[draft specification](https://modelcontextprotocol.io/specification/draft),
and the
[changelog](https://modelcontextprotocol.io/specification/draft/changelog)
lists every change against `2025-11-25`.

## Tell us what breaks

Four weeks remain until the specification is final. Here is what actually
helps:

1. Install a beta in a branch of your server and run your real traffic
   against it, not just the happy path.
2. If you operate behind a gateway or load balancer, serve the stateless
   path and see whether your routing still needs anything the protocol no
   longer provides. In TypeScript that path is `createMcpHandler`; in
   Python, the v2 HTTP app answers both protocol revisions from one
   endpoint.
3. File what you find. SDK problems go to the
   [Python SDK](https://github.com/modelcontextprotocol/python-sdk/issues)
   or
   [TypeScript SDK](https://github.com/modelcontextprotocol/typescript-sdk/issues)
   issue trackers; problems with the protocol itself go to the
   [specification repository](https://github.com/modelcontextprotocol/modelcontextprotocol/issues).

Public APIs may still change between the betas and the stable releases, so
pin exact versions. For Python that is the `==2.0.0b1` pin above; for npm,
add `--save-exact` so the install writes the resolved version rather than a
range. The stable v2 releases are targeted alongside the final
specification, and your feedback in the next four weeks is what they will be
built from. For implementation questions, the relevant
[Working Group](https://modelcontextprotocol.io/community/working-interest-groups)
channel in the
[contributor Discord](https://modelcontextprotocol.io/community/communication#discord)
is the fastest path to an answer.
