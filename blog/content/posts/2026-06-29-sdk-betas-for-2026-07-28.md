Beta releases of the official
[Python](https://github.com/modelcontextprotocol/python-sdk) and
[TypeScript](https://github.com/modelcontextprotocol/typescript-sdk) SDKs are
now available with support for the
[`2026-07-28` specification release candidate](/posts/2026-07-28-release-candidate/).
These betas are the validation milestone in the window between the release
candidate lock on May 21 and the final specification on July 28, 2026. If you
maintain an MCP server or client, this is the right time to build against the
new protocol version and tell us what breaks.

Both betas are opt-in. The stable v1.x lines remain the recommended versions
for production, and installing either SDK without an explicit version pin
still resolves to v1.x.

The [Go](https://github.com/modelcontextprotocol/go-sdk) and
[C#](https://github.com/modelcontextprotocol/csharp-sdk) SDKs will also ship
beta releases with `2026-07-28` support. Watch their repositories for the
announcements.

## Installing the betas

Python:

```bash
pip install mcp==2.0.0b1
# or
uv add "mcp==2.0.0b1"
```

TypeScript ships v2 as split packages, one for servers and one for clients,
plus thin middleware adapters for Node.js, Express, Hono, and Fastify:

```bash
npm install @modelcontextprotocol/server@beta
npm install @modelcontextprotocol/client@beta
```

## What the betas support

Both SDKs implement the headline changes of `2026-07-28`:

- **The stateless protocol core**
  ([SEP-2575](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2575),
  [SEP-2567](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2567)).
  The `initialize` handshake and `Mcp-Session-Id` are gone on the new path.
  Each request is self-describing, servers expose capabilities through
  `server/discover`, and any server instance can handle any request. Client
  version negotiation is automatic: clients probe the new path and fall back
  to the `initialize` handshake when they reach a server that only speaks
  `2025-11-25` or earlier.
- **Multi round-trip requests**
  ([SEP-2322](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2322)).
  Tool calls can return `InputRequiredResult`, and clients can answer the
  input requests and retry the call with the echoed request state, with no
  long-lived stream in between.
- **Routable transport headers**
  ([SEP-2243](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2243)).
  The Streamable HTTP implementations send and enforce the `Mcp-Method` and
  `Mcp-Name` headers.
- **Authorization hardening**. The OAuth clients validate the `iss`
  authorization-response parameter per
  [RFC 9207](https://www.rfc-editor.org/rfc/rfc9207)
  ([SEP-2468](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2468)),
  declare `application_type` during Dynamic Client Registration
  ([SEP-837](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/837)),
  accumulate scopes on step-up re-authorization
  ([SEP-2350](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2350)),
  and bind registered credentials to the authorization server that issued
  them
  ([SEP-2352](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2352)).
- **Updated error codes**
  ([SEP-2164](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2164)).
  A missing resource now returns the JSON-RPC standard `-32602` instead of
  the MCP-custom `-32002`.
- **Deprecation annotations**
  ([SEP-2577](https://github.com/modelcontextprotocol/modelcontextprotocol/pull/2577)).
  Roots, sampling, and `logging/setLevel` APIs are marked deprecated in both
  SDKs. The deprecations are advisory only; wire behavior is unchanged, and
  everything keeps working for sessions negotiating `2025-11-25` or earlier.

## Python specifics

The Python beta also splits the wire types into a standalone
[`mcp-types`](https://pypi.org/project/mcp-types/) package that depends only
on `pydantic` and `typing-extensions`, so tooling and lightweight clients can
validate MCP traffic without pulling in the transport stack. Server
middleware now takes a `(ctx, call_next)` signature, and a new
`OpenTelemetryMiddleware` emits spans with the OpenTelemetry GenAI
semantic-convention attributes.

The
[migration guide](https://github.com/modelcontextprotocol/python-sdk/blob/main/docs/migration.md)
covers the full list of breaking changes from v1.x.

## TypeScript specifics

The TypeScript v2 packages replace the single `@modelcontextprotocol/sdk`
package. Tool and prompt schemas use
[Standard Schema](https://standardschema.dev/), so you can bring Zod v4,
Valibot, ArkType, or any compatible library. The SDK runs on Node.js, Bun,
and Deno.

The [v2 API documentation](https://ts.sdk.modelcontextprotocol.io/v2/) covers
the new package layout and the differences from v1.x.

## What happens to v1.x

Nothing changes for production users today. The v1.x lines of both SDKs
remain the recommended versions until the stable v2 releases ship alongside
the final specification, and they will continue to receive bug fixes and
security updates for at least six months after v2 ships.

## Tell us what breaks

Four weeks remain until the specification is final. Use them to validate
these changes against real workloads. Public APIs may still change between the betas and the stable
releases, so pin exact versions and expect to update.

If you find a problem in an SDK, open an issue in the
[Python SDK](https://github.com/modelcontextprotocol/python-sdk/issues) or
[TypeScript SDK](https://github.com/modelcontextprotocol/typescript-sdk/issues)
repository. If the problem is with the specification itself, open an issue in
the [specification repository](https://github.com/modelcontextprotocol/modelcontextprotocol/issues).
For implementation questions, the relevant
[Working Group](https://modelcontextprotocol.io/community/working-interest-groups)
channel in the
[contributor Discord](https://modelcontextprotocol.io/community/communication#discord)
is the fastest path to an answer.
